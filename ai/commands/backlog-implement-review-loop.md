Run one fresh-context pass of the backlog implementation/review loop for `$ARGUMENTS`. This command owns pass selection, implementation/review authority, fixes, commits, integration, and handoff. The `backlog-source-workflow` skill owns provider-agnostic source resolution, item/state normalization, scheduling, provider methods, and durable provider state.

## Source and scope contract

Load and follow `backlog-source-workflow` before interpreting `$ARGUMENTS`: read its provider-neutral contract first, resolve every provider kind represented by explicit sources, then load one matching provider heading per kind. Use `selectNext`, `readItem`, the selected claim authority, heartbeat/release, and guarded provider writes rather than duplicating scheduling or lock rules here. Pass the original argument string unchanged to `resolveSource`; explicit sources and selectors retain argument order. For item selectors, precedence is stable provider ID, exact title, then exact description; dependency readiness is evaluated before selector ordering.
If an explicit source or selector cannot resolve, report that argument and candidates; never fall back to another or derived source.

A source-only input resolves the whole provider collection. It is a scheduling scope, not a whole-source review boundary: on every fresh invocation, rebuild the complete dependency graph and recompute provider blockers, dependency-gated chains, terminal state, review markers, resumable progress, and active/expired claims before selection. Dependencies are a hard gate, but an unfinished prerequisite represented by another defined backlog item is dependency-gated (`ready after <item>`), not `BLOCKED`. `selectNext` may move past an actively claimed item only to another independent item in the same dependency-ready wave; it never selects that item's dependent. Unclaimed or expired-claim in-progress work ranks before new work. Explicit item IDs, titles, descriptions, and ranges remain the exact item scope and retain original argument/source order.

The default implementation and review boundary is one provider item. An explicit `review-group:<provider-native-selector>` is passed unchanged to `reviewBoundary`; only the provider-authorized exact member items may be reviewed, and the durable group marker must name those members and accumulated commit state. A milestone, epic, parent, or other multi-item review is permitted only when the caller explicitly supplies that selector and the skill can resolve and persist it. Never infer a ReviewGroup from source-only scope, shared labels, or adjacency. Whole-source scheduling therefore cannot be mistaken for whole-source review.

Provider details are not duplicated here. A Backlog.md source supplied from any Git worktree resolves to the unambiguous primary/control checkout, and every provider read/write runs there; implementation worktrees are code-only. Backlog.md uses fenced same-host item claims around CLI mutations when available and coordination-only claims around MCP writes otherwise; `@<assignee>` remains visibility. Remote GitHub uses `gh` plus fenced same-host item claims. Loose Markdown uses one fenced source claim and `replace-file` CAS. Linear and any future provider without native mutation fencing default to a canonical same-host `local-coordination` item lease and direct first-party provider writes with pre/post checks. Provider workflow state remains authoritative; the local lease store contains coordination only. Never treat an assignee/status/comment/branch as a claim or describe local coordination as provider-side/cross-host fencing. If a canonical Backlog.md control checkout cannot be identified or safely used, select `WAIT`; never use a feature-worktree copy as fallback provider state.

## One-pass invariant

Every invocation executes exactly one of these states:

1. `IMPLEMENT`: claim and complete exactly one next coherent implementation task for the current item. Include inseparable production code, required callsites, fixtures/migrations, and tests, but never a second refined task. Verify it, commit only its task-related change, persist its progress through the skill, release the claim, and hand off the next task. Never review while an implementation task remains.
2. `REVIEW`: only after the current item's explicit implementation tasks and acceptance criteria are complete, claim and review that one item's entire accumulated implementation together. A batch is allowed only for an explicit durable `ReviewGroup`; claim every exact member or review none. Write one marker per member. A clean or fixed review does not schedule another review pass for the same exact commit state.
3. `BLOCKED`: only after a claimed pass has exhausted safe work and a provider-authored blocker, missing/cyclic dependency reference, unavailable external dependency/integration not represented by a defined backlog item, evidence-backed external/human decision, or unsafe ambiguity that cannot be resolved without such a decision genuinely prevents it. An unfinished prerequisite represented by another defined backlog item is dependency-gated (`ready after <item>`), not `BLOCKED`; persist the blocker and unblock condition through the skill, then release the claim. Do not use blockers for ordinary code/test failures or active claims.
4. `WAIT`: execute no implementation/review/archive work when `selectNext` reports that every unfinished root is genuinely blocked or actively claimed, including dependency-gated chains held by those roots. Report genuine blockers and claim owners/expiries; do not steal a claim or select a dependent.
5. `ARCHIVE`: after all scoped items have complete implementation and valid review markers, claim the archive target and use provider-owned archive operations where required. Archive is not a review boundary.

Do not alternate implementation and review passes. An item with remaining tasks selects `IMPLEMENT`; an item with all tasks done but no valid marker selects `REVIEW`; an item with a valid provider/human/missing-reference/cycle blocker is dependency-ineligible, while an ordinary dependent of an unfinished defined backlog item is merely dependency-gated; and an item with an active claim is unavailable to other sessions. Independent roots in the same ready wave remain claimable. When all scoped work is terminal, select `ARCHIVE` only if archive remains required and authorized; otherwise report `BACKLOG COMPLETE AND ARCHIVED`. When unfinished roots are all genuinely blocked or actively claimed, select `WAIT`. Never reach into a later dependency wave to manufacture work.

## Claim lifecycle

Before isolation, implementation edits, code-diff review, blocker mutation, integration, or archive:

1. Preflight exact pass reads/writes and select the strongest available capability: native fencing, helper-fenced `item-claim`/`source-claim`, then helper `local-coordination`. Unsupported provider fencing automatically falls back to local coordination; `none` becomes `WAIT` only when required provider operations or a canonical local lease are unavailable. Label every weaker pass `LOCAL COORDINATION (UNFENCED)` and pass `--coordination-only` to `backlog-claim acquire`.
2. Refresh provider scheduling/dependency state and local claim state. Generate unique claim/session/worker-attempt IDs, derive the canonical provider/source/item resource, then acquire the exact work key: `implement:<item>:<task>:<version>`, `review:<item-or-group>:<commit-set>`, or `archive:<target>:<version>`. For Backlog.md, resolve the canonical control checkout before acquisition, then project `@<assignee>` and durable `in-progress` state there through guarded CLI or checked MCP. Complete and verify this canonical checkpoint before creating or switching to implementation isolation: `claim -> canonical in-progress checkpoint -> isolation`. For Linear and similar providers, assign an unassigned item to the authenticated provider user after acquisition; preserve a different assignee and stop. Assignment never supplies claim capability. Reread ambiguous results by exact claim ID and confirm the receipt's guarantee matches the selected mode. For a `ReviewGroup`, acquire members in stable order under the same guarantee; on definite conflict release acquired members, while ambiguous acquisition remains held until resolved/expired. Select `WAIT` rather than reviewing a partial group.
3. Revalidate dependencies after acquisition and at heartbeat/checkpoint boundaries. Heartbeat by current claim ID/token/revision before half the lease elapses and around long operations. Fenced provider mutations use `backlog-claim exec`, `replace-file`, or a native provider fence. Coordination-only mutations run directly through first-party tooling: heartbeat and validate the exact lease plus provider eligibility/version immediately before each call, then reread both immediately afterward. Stop on expiry, replacement, assignee conflict, file/provider change, failure, ambiguity, or reopened dependency. Never run Linear MCP/tool calls through `backlog-claim exec`.
4. Persist the coherent pass checkpoint under the valid claim, retain provider and claim receipts, then release by exact claim ID/token/current revision. A normal one-pass handoff releases while leaving unfinished work `in-progress`, so a fresh session acquires a new claim immediately. Failed/ambiguous checkpoint or release does not advance scheduling.

For repository-backed provider state, use a short same-host provider/repository mutation transaction lock around each canonical provider write, provider-state commit, or Git integration. This transaction lock is distinct from and shorter than the pass `WorkClaim`; refresh shared provider/Git state after acquiring it, release it immediately afterward, and never hold it through coding, tests, or review. Independent ready items may therefore implement concurrently in separate code-only worktrees while shared provider mutations and integration serialize. Unresolved contention or unsafe shared state selects `WAIT`. The transaction lock does not provide cross-host fencing.

The active agent owns the claim even when it delegates bounded investigation or verification. A child never independently claims the same item.

## IMPLEMENT pass

Validate every explicit local path before isolation. A missing explicit path may substitute only one clearly adjacent same-directory or moved/renamed-basename candidate, and must report the substitution; otherwise stop before any edit rather than falling back to another or derived source.
Before remote-only isolation, preflight evidence, status transitions, completion/archive, and the selected claim mode. Remote location alone does not imply one guarantee: GitHub uses fenced helper-backed `gh`, while Linear defaults to native fencing when available and `local-coordination` otherwise. Proceed under local coordination when provider operations and the canonical local lease are available, label the pass `LOCAL COORDINATION (UNFENCED)`, and preserve another assignee. Stop before isolation/edits only when capability is truly `none`. For Backlog.md, create or switch to isolated code work only after claim acquisition, dependency revalidation, and the canonical `in-progress` checkpoint; never run provider reads or writes against the implementation worktree. Preserve unrelated dirty/untracked changes. If isolation fails, stop before code changes and checkpoint the safe canonical state before release. Then inspect the complete selected item, refined tasks, criteria, dependencies, progress, code patterns, and callsites.

Select exactly the first open task named by the claim. Run the smallest targeted verification, fix in-scope failures, and rerun. With the current ownership epoch, persist task, implementation commit, exact command/result, progress state, next task, and remaining criteria to canonical provider state through the selected fenced path or coordination-only pre/post-check sequence, then release. Every pass verifies that canonical checkpoint before releasing its one-pass lease. Failed or ambiguous writes retain the code commit and return `NEXT CONTEXT REQUIRED` with state repair.

The loop driver is `/loop`, which supplies a fresh context for the next invocation. Do not run a forever-loop or invent a narrowed continuation argument. Any copy-pasteable continuation must invoke this command with the original `$ARGUMENTS` unchanged; durable state and whole-collection scheduling select the next target.

## REVIEW pass and right-sized review

Before `REVIEW`, acquire item/member claims, refresh provider state plus local claim state, and verify each claim ID/token/revision and dependency remains current. Resolve one unambiguous implementation target per item from provider links/markers and history. Missing/ambiguous association becomes `BLOCKED`: persist evidence through the selected claim mode, release, and inspect no unrelated code. Provider metadata supplies intent only, never the diff. Review exact resolved ranges and every acceptance/error path at proportionate depth:

- For a small, tightly scoped item confined to one behavior/subsystem, perform a direct lightweight correctness/acceptance review. Read the complete accumulated diff, check relevant failure paths and tests, run targeted verification, and fix valid findings at the source.
- Load and apply the full `implementation-review` skill for any materially large or complex change; independent subsystems or other cross-subsystem changes; auth, security, or privacy-sensitive behavior; schema, migration, or data-integrity work; concurrency, transactions, or state coordination; public API or compatibility changes; meaningful performance risk; or an explicitly requested deep review. Apply all relevant rubric areas, not just style, and use the command's fix, marker, and integration authority.

Never skip correctness or acceptance verification merely because the lightweight path is selected. Do not load the heavy skill automatically for every quick loop review. Do not manufacture findings; valid fixes are in-scope review work, and review-fix commits are included in the same review pass.

After review, run targeted verification. Under each current ownership epoch, persist item-local `reviewed:` evidence through the selected fenced path or coordination-only pre/post-check sequence, naming exact implementation and review-fix commits. An explicit `ReviewGroup` marker names every member/commit state while all claims remain active; retain item markers. A later relevant code/acceptance change invalidates the marker.

If review finds substantial missing implementation, persist the exact next `IMPLEMENT` task under the current ownership epoch, then release. Otherwise record completion/integration only after all tasks, acceptance, review evidence, and provider state are durable under that epoch, then release. Do not schedule a separate review solely for a review-fix commit.

## Blockers, verification, and delegation

Before recording any notable genuine blocker, exhaust safe repository work and targeted verification, then use at most one evidence-backed oracle consultation for that pass, batching related questions. The oracle is advisory and read-only with no code, backlog, provider, commit, or integration mutation authority; the active agent must independently verify any recommendation and owns every mutation. Do not consult for routine choices or ordinary failures; record accepted or rejected reasoning with the blocker evidence.
A blocker marker is durable only after repository context, safe implementation, targeted checks, and stale/missing causes are exhausted and the cause is a provider-authored blocker, missing/cyclic reference, unavailable external dependency/integration not represented by a defined backlog item, or evidence-backed external/human decision. Include reason, evidence, attempts, and unblock condition; write through the selected fenced path or coordination-only pre/post-check sequence, then release. Re-enter when code, dependencies, backlog text, or evidence changes. An active claim is `WAIT`, not a blocker; an ordinary unfinished prerequisite represented by another defined backlog item is dependency-gated (`ready after <item>`), not a blocker. Report unrelated failures/dirty work separately.

## Integration and archive authority

Obey repository `CLAUDE.md`/`AGENTS.md`/backlog integration configuration. Otherwise use pull-request flow when push access, protection, or remote ownership requires it; when ambiguous, use local merge. Local merge acquires the short repository/provider mutation transaction lock, refreshes shared Git/provider state, integrates verified completed code into local `main`, records canonical provider state, then releases the transaction lock and cleans up its temporary code worktree without pushing. Pull-request flow pushes only this task branch and opens a PR with `gh`, without merging it; report the URL and recommend `/pr-babysit [reviewer]`. Never integrate unrelated work. When all explicitly scoped items (or all members of an explicit ReviewGroup) are complete, reviewed, integrated, and durably recorded, archive writable local sources only through the provider adapter and existing convention. Source-only completion requires the entire resolved collection to have no remaining open work; it does not authorize reviewing or archiving unrelated sources.

## Handoff and final statuses

Start every response with exactly one status line: `NEXT CONTEXT REQUIRED` or `BACKLOG COMPLETE AND ARCHIVED`.

Then report source order, original arguments, scheduled item, pass state (`IMPLEMENT`, `REVIEW`, `BLOCKED`, `WAIT`, or `ARCHIVE`), exact task/group, claim guarantee (`FENCED` or `LOCAL COORDINATION (UNFENCED)`), claim ID/owner/work key/revision/expiry and acquire-heartbeat-release receipts, provider pre/post receipts when unfenced, commits/files, targeted verification, review depth/findings/fixes, durable markers, integration/archive, skipped blockers, unrelated failures/dirty changes, and exact next target. For `WAIT`, list every blocking/claimed root and gated chain. Continue with unchanged `$ARGUMENTS`. Use `NEXT CONTEXT REQUIRED` for unfinished, unreviewed, blocked, actively claimed, unintegrated, non-durable, ambiguous ownership/write/release, or required archive state. Use `BACKLOG COMPLETE AND ARCHIVED` only when all scoped work is verified, reviewed at exact commits, durable, integrated, archived where required, and unclaimed.
