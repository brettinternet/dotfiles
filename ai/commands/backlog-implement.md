Implement the selected backlog work from `$ARGUMENTS` through its refined tasks, verify it independently, record durable progress, commit only task-related changes, and integrate each completed item using the repository's declared flow. This command owns implementation scope and authority; `backlog-source-workflow` owns provider resolution, scheduling, and durable provider operations. Accumulated implementation review belongs to `backlog-implement-review-loop` or `review-implementation`, not this command.

## Source and scope

Load `backlog-source-workflow` before interpreting `$ARGUMENTS`: read its provider-neutral contract first, resolve every explicit provider kind in supplied source order, then load one matching provider heading per resolved kind. Use `Source`, `SchedulingScope`, `ItemState`, and `WorkClaim` plus `resolveSource`, `discover`, `selectNext`, `readItem`, claim/heartbeat/release, selected-mode `writeState`/`recordProgress`, and `archive`. Use the selected provider's native reads/writes and claim authority; do not resolve a `ReviewGroup` or reproduce provider rules here.

Preserve explicit source and selector order. Build the complete dependency graph before selection; an incomplete prerequisite represented by another defined backlog item is a hard scheduling gate, not a reason to mark the dependent `blocked`; report it as `ready after <item>` and wait for the prerequisite to reach terminal state. Missing references, cycles, provider-authored blockers, and evidence-backed external/human decisions remain genuine blockers. Selector order wins within a dependency-ready wave, while provider priority/ordinal applies only within a source-only wave. Derive exactly one repository source only when no source is explicit. Never fall back from an unresolved explicit source; a missing local path may substitute only one clearly adjacent same-directory or moved/renamed-basename candidate, and the substitution must be reported.

Explicit selectors identify exactly the items to implement. With only a source, select the first claimable item from the earliest dependency-ready wave. An active claim may be bypassed only for an independent item in that same wave, never for its dependent; unclaimed or expired-claim in-progress work ranks before new work. If all scoped roots are terminal, genuinely blocked, or actively claimed, perform no implementation and report complete/blocker/claim-owner-expiry state. The default implementation boundary is one provider item.

Before implementation context, isolation, delegation, or edits, preflight authorized provider reads/writes and select the strongest available claim guarantee: native fencing, helper-fenced `item-claim`/`source-claim`, then helper `local-coordination`. Unsupported provider fencing never stops otherwise authorized work; derive a canonical provider/source/item resource, acquire it with `backlog-claim acquire --coordination-only`, and report `LOCAL COORDINATION (UNFENCED)`. Capability `none` stops work only when required provider operations or a canonical local lease are unavailable. Refresh provider state plus claim state and resolve `<next-task>` before acquisition: use the first unchecked stable ID, assign stable IDs by existing order to an equivalent task list, or use fallback ID `T1` when no usable list exists and scope is unambiguous. Generate unique claim/session/worker-attempt IDs, then acquire `implement-item:<item-id>:<next-task>:<provider-version>`. Backlog.md/GitHub/loose Markdown use their fenced helper paths when available. Linear and other unfenced providers default to local coordination; after acquisition, assign an unassigned item to the authenticated provider user for visibility and preserve a different assignee. Before every direct first-party mutation, heartbeat and validate the exact lease plus provider eligibility/version; reread both immediately afterward and stop on lease loss, assignee conflict, provider change, failure, or ambiguity. Never run coordination-only or Linear MCP/tool calls through `backlog-claim exec`, call assignment a claim, or imply provider-side/cross-host fencing. Reread ambiguous acquisition by exact claim ID and require the receipt's guarantee to match the selected mode; revalidate dependencies after acquisition, then read the full specification/tasks/acceptance/progress/code/callsites, repair or synthesize the checklist when possible, and create isolated work.

## Implementation and durable progress

Implement every available refined task in the claimed item in order. Select exactly the first unchecked task under `### Implementation tasks` when present; checked tasks are complete and must not be repeated unless explicitly reopened by a later review. If the checklist is absent or malformed, use the pre-claim equivalent-list ID or fallback ID `T1`, then derive the smallest bounded task from the unambiguous item specification and normalize the checklist when possible. One task includes inseparable production code, callsites, fixtures/migrations, and proving tests. The holder revalidates dependencies and heartbeats by claim ID/token/current revision before half the lease elapses and around long checks/integration. After each task:

1. Run the smallest targeted verification; fix in-scope failures and rerun.
2. Commit the coherent implementation without unrelated work.
3. When the checklist is writable, change exactly the selected task from `[ ]` to `[x]`, preserving its stable ID and text. Do this only after verification and the implementation commit; failed checks leave the task unchecked. For a fallback task, persist its stable ID and completion through the provider's existing supported progress representation until the checklist can be normalized.
4. With the current ownership epoch, persist the checkbox or fallback task state, status, completed task/commit, exact verification, next task, and remaining acceptance criteria through the selected provider checkpoint.
5. Persist every progress/status/blocker/completion/archive mutation through the selected claim mode: `backlog-claim exec` for one fenced Backlog.md/GitHub CLI mutation, `backlog-claim replace-file` for one fenced loose-Markdown source CAS, or the local-coordination lease/provider pre/post-check sequence for one direct first-party mutation. Retain claim and provider receipts and never label coordination-only writes as fenced.

If the canonical checklist is absent or malformed, do not stop solely for that formatting defect. Repair or synthesize it under the current claim when possible; otherwise use a deterministic bounded fallback only when scope is unambiguous and record the fallback through existing provider progress state. If the code commit succeeds but the checkbox or provider checkpoint is ambiguous or fails, retain the claim and commit, repair the durable state, and do not report the task complete. Genuine scope or authority ambiguity still stops the pass.

The active agent owns the claim; children do not. Heartbeat before half the lease elapses and around long work; guarded CLI execution renews while its child runs, while local coordination requires an explicit heartbeat immediately before each provider mutation. Expiry, replacement, provider/file-version conflict, assignee conflict, or reopened dependency stops provider mutation/integration. Ambiguous ownership or coordination-only write outcome is resolved by rereading the exact claim ID and provider state, never adoption. Failed write-back retains the code commit and claim until repaired/expired. Persist a genuine blocker or coherent early-handoff checkpoint under the current epoch, then release with current token/revision; `in-progress` remains immediately resumable by a fresh claim.

## Independent verification

When all implementation tasks are durably complete—preferably represented by checked task boxes—and acceptance criteria appear complete, leave the claimed item verification-pending and run one final `verifier` from criteria, commits, files, and evidence. Every `FAIL`/`UNVERIFIED` remains implementation work. Mark complete only after all pass and provider evidence is durable through the current claim ID/token/revision guard.

Delegate implementation only for materially substantial independent branches; keep shared interfaces, decisions, task-coupled tests, verifier synthesis, and integration in the active agent. Before recording a notable genuine blocker, gather evidence and use at most one read-only oracle consultation for that pass. Independently verify its recommendation; the active agent retains all mutation authority.

## Integration and completion

Resolve integration from repository `CLAUDE.md`, `AGENTS.md`, or backlog configuration. Otherwise use pull-request flow when push access, branch protection, or remote ownership requires it; when ambiguous, use local merge.

- `local-merge`: merge completed verified work into local `main`, clean up its temporary worktree, and do not push.
- `pull-request`: push only the task branch, open a PR with `gh`, do not merge it, report the URL, and recommend `/pr-babysit [reviewer]`.

An item is complete only when every implementation task is durably complete, acceptance criteria, targeted verification, independent verifier, integration, provider completion through the selected claim mode, and `releaseClaim` succeed. Prefer checked task boxes; a provider-equivalent task checkpoint is acceptable when checklist repair is still outstanding. Source-only scope completes only with no open item or active claim. Otherwise report `NEXT CONTEXT REQUIRED` with the next task, genuine blocking roots, dependency-gated chains, and active claim owners/expiries plus gated chains. Use `BACKLOG COMPLETE AND ARCHIVED` only after all scoped work/archive is complete.

Report status first, then source order, selected item/wave, tasks, claim guarantee (`FENCED` or `LOCAL COORDINATION (UNFENCED)`), claim ID/owner/work key/revision/expiry and receipts, provider pre/post receipts when unfenced, commits/checks/verifier, durable writes, integration/archive, next target, unrelated changes/failures, blockers, and active claims. Continue with unchanged `$ARGUMENTS`.
